---
title: "The `init` skill"
description: "Bootstrap a repo's `.claude/` setup by inventorying the codebase, matching it to a starter-set archetype, and delegating every write to `core:evolve`."
---

`/core:init` is the entry point for setting up Claude Code in a repo you haven't configured yet. It reads the repo, matches it to a known archetype, proposes the smallest useful set of artifacts (≤3), gets your approval, then hands off entirely to `core:evolve`. It writes nothing itself.

## When to use

Invoke `/core:init` when you want to:

- Set up `.claude/` in a repo for the first time.
- Get a curated starter set of skills, sub-agents, or hooks matched to your stack.
- Avoid guessing which artifacts would actually be useful here.

<Tip>
No arguments needed. The empty-args path is the canonical use — `/core:init` with nothing after it triggers a fully automated inventory and proposal with zero clarifying questions.
</Tip>

## How it works

<Steps>
  <Step title="Detect intent">
    Reads `$ARGUMENTS`. Empty args proceed immediately — no clarifying question. A vague keyword (e.g. `bootstrap`, `setup`) triggers one focused question to narrow the scope. A concrete constraint (e.g. `skills only`, `focus on testing`) is adopted as a Phase 3 filter and the skill proceeds.
  </Step>
  <Step title="Repo inventory">
    Delegates a full read to `explore:explore`: languages, build tool, test framework, CI config, existing `.claude/` artifacts, `AGENTS.md` / `CLAUDE.md` presence, and plugin manifests. No source files are read from the main thread — delegation only.
  </Step>
  <Step title="Target detection">
    Runs `Glob plugins/**/.claude-plugin/plugin.json`. If one or more plugin directories are found, asks you to pick the write target (one option per plugin, plus `Local .claude/`). If none are found, silently targets `.claude/` in the repo root.
  </Step>
  <Step title="Preliminary classification">
    Calls `core:skills-list` to get the installed-artifact catalog for the selected target, then matches the inventory against the archetype starter sets. Each candidate is classified as `create`, `update`, or `reuse` using the 70% overlap threshold from `core:evolve`'s decision matrix.
  </Step>
  <Step title="Targeted brainstorm (optional)">
    Presents a four-option prompt: deepen the inventory, widen with a brainstorm, converge to a proposal, or stop. One round by default. Selecting `Converger` or skipping straight to convergence passes the Phase 4 result through unchanged.
  </Step>
  <Step title="Final classification">
    If Phase 4 returned new ideas, re-runs the classification with them folded in. Otherwise this is a pass-through of the Phase 4 result.
  </Step>
  <Step title="Batch approval">
    Presents the full proposal as a numbered list (≤3 artifacts), each annotated with its classification and target. Three choices: `Accept`, `Edit` (per-artifact override then re-present), or `Reject` (clean exit). If more than 3 artifacts would be useful, only the top 3 are shown — the rest are deferred to a follow-up `/core:init` run.
  </Step>
  <Step title="Delegate to core:evolve">
    Fires a single `Skill({ skill: "core:evolve", args: "..." })` call with the consolidated approved proposal. After that, `/core:init` is done. `core:evolve` owns the write path and produces the only execution report.
  </Step>
</Steps>

## Repo archetypes

`/core:init` matches the inventory against four built-in archetypes. Each archetype maps a detection signal to a ≤3-item starter set.

| Archetype | Detection signal | Starter set |
|---|---|---|
| Claude Code marketplace | `plugins/**/.claude-plugin/plugin.json` exists | `release-notes` skill + `manifest-sync` hook + optional `skill-author-check` sub-agent |
| Node/TS monorepo | `package.json` + workspace file or `packages/` directory | `lint-fix` skill + `pr-check` sub-agent + optional pre-commit format hook |
| Python library | `pyproject.toml` at repo root | `pytest` skill + `release` skill + optional coverage-report hook |
| Generic fallback | None of the above | Recommends invoking `/core:evolve` directly — no artifacts auto-proposed |

<Note>
The ≤3 cap is structural, not advisory. `/core:init` never proposes a fourth artifact in a single run. If your repo warrants more, run `/core:init` again after the first batch is installed.
</Note>

## Composition model

`/core:init` is a pure orchestrator. It delegates every specialized concern to an existing skill:

| Concern | Delegated to |
|---|---|
| Codebase reading | `explore:explore` |
| Installed-artifact catalog | `core:skills-list` |
| Ideation (if requested) | `core:brainstorm` |
| Artifact authoring | `core:evolve` → `skill-architect` / `subagent-architect` / `hooks` |

It never re-implements analysis, ideation, or file authoring. If you need any of those independently, invoke the relevant skill directly.

## Hard rules

- **Never writes artifact files directly** — `core:evolve` owns the entire write path.
- **Never reads source files from the main thread** — all repo inspection is delegated to `explore:explore`.
- **Never exceeds the ≤3-artifact starter set** without an explicit user request.
- **Never writes any brief or summary file to disk** — `core:evolve`'s execution report is the single record.
- **Always ends each phase on user input or a deterministic signal** — no silent phase-skipping.
