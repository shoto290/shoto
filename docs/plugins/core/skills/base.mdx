---
title: "The `base` skill (reference)"
description: "Shared foundation rules loaded by all core skills and architects. Not user-invocable."
---

`base` is the portable foundation that all other `core` skills and sub-agents extend from. It is **not** a skill you invoke directly — it is loaded by reference, the same way a parent class is imported by a subclass.

<Note>
`base` has `disable-model-invocation: true`. It never auto-triggers from user prompts and does not appear in `/help` as an invocable command. It is loaded explicitly via a Markdown link from sibling skills, or via the `skills:` preload field in sub-agent frontmatter.
</Note>

## Purpose

`base` keeps one canonical statement of principles and behavioral guidelines instead of duplicating them in every artifact. Any skill or sub-agent that extends `base` automatically inherits:

- The **SIMPLE** engineering principles (Simple, Intentional, Measurable, Pragmatic, Layered, Envisioned).
- Four behavioral guidelines: Think Before Coding, Simplicity First, Surgical Changes, Goal-Driven Execution.
- Common workflows: inventory before proposing, confirmation before destructive git operations, validation gate before returning.
- Naming conventions reference (kebab-case files, `name:` must match path, headings in Title Case).

## How skills extend `base`

Add this quoted line at the top of a `SKILL.md` body, immediately after frontmatter:

```markdown
> Apply the rules from [core:base](../base/SKILL.md) in addition to those below.
```

This relative path only resolves for plugin-scoped skills that live as siblings of `base/` — i.e., other skills inside the `core` plugin. Do not add this line to project-scoped or user-scoped skills.

## How sub-agents extend `base`

Add `base` as the first entry in the `skills:` preload field in frontmatter:

```yaml
skills:
  - base
  - skill
```

The full content of `base/SKILL.md` is injected into the sub-agent's context at startup.

<Warning>
The `skills: [base]` pattern only resolves when the sub-agent lives inside the `core` plugin (or a plugin that has a sibling `base/` directory). For sub-agents in `.claude/agents/` or `~/.claude/agents/`, the `base` skill is unavailable — it is not copied into those scopes.
</Warning>

## SIMPLE principles (summary)

| Letter | Principle |
|---|---|
| S | Simple — favor the simplest solution. Less code, fewer abstractions. |
| I | Intentional — every line exists for a reason. No speculative features. |
| M | Measurable — changes must have observable impact. |
| P | Pragmatic — ship what works today. Proven patterns over clever ones. |
| L | Layered — each change is a stable, shippable layer. |
| E | Envisioned — short-term decisions align with the long-term goal. |

## Delegation targets

When a skill or sub-agent needs one of the capabilities below, delegate to the canonical skill via `Skill({ skill: "<name>", args: "<topic>" })` instead of re-implementing it. Re-implementation duplicates logic, drifts over time, and bypasses the careful structure of the canonical skill.

| Intent / capability | Canonical skill |
|---|---|
| Codebase exploration — understand code, find patterns, locate components, trace flows, audits | `explore:explore` |
| Create or update a skill | `core:skill` |
| Create or update a sub-agent | `core:subagent` |
| Create or update a hook | `core:hooks` |
| Brainstorm an idea before planning | `core:brainstorm` |
| Plan a coordinated multi-artifact change (skill + sub-agent + hook) | `core:evolve` |
| Make a single git commit | `git:commit` |
| Open a pull request | `git:create` |
| Rebase the current branch onto main | `git:rebase` |
| Review PR comments / process review feedback | `git:review-comments` |
| Apply review fix decisions | `git:review-fix` |
| Review the current workspace diff | `git:review-diff` |
| Discover which already-installed skill fits a task | `core:skills-suggest` |
| List skills available in the current context (global / project / marketplace) | `core:skills-list` |

```text
Skill({ skill: "explore:explore", args: "trace the auth flow from login endpoint through middleware to session store" })
```

<Note>
If you author a skill that needs one of these capabilities, name the delegation explicitly in your body — a one-line note in the relevant operational section. Do not re-implement.
</Note>
