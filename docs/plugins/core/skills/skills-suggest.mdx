title: "The `skills-suggest` skill"
description: "Owns the canonical intent → skill map for the marketplace. Routes a free-form intent to the best-fit already-installed skill(s) with justification."
---

`/core:skills-suggest <intent>` routes a natural-language description of what you want to accomplish to the right already-installed skill. It owns the canonical intent → skill map for the marketplace, cross-checks it against what is actually installed via [`core:skills-list`](/plugins/core/skills/skills-list), and returns up to three ranked candidates, each with a one-line justification, a copy-pasteable invocation, and a boundary statement.

<Note>
  `core:skills-suggest` routes among skills you already have installed. To install a new skill, use `find-skills`. To create a new skill from scratch, use [`core:skill`](/plugins/core/skills/skill) or [`core:evolve`](/plugins/core/skills/evolve).
</Note>

## Overview

With dozens of installed skills, picking the right one is non-trivial. This skill removes that decision cost by owning the canonical map of intent → skill for the marketplace, cross-checking it against what is actually installed locally, and returning a ranked shortlist. Because it calls `core:skills-list` internally, it never recommends a skill you don't have.

## Delegation map

This is the single source of truth for capability → canonical skill. When a skill or sub-agent needs one of these capabilities, delegate to the listed skill rather than re-implementing.

| Intent / capability | Canonical skill |
| :--- | :--- |
| Codebase exploration — understand code, find patterns, locate components, trace flows, audits | `explore:explore` |
| Create or update a skill | `core:skill` |
| Create or update a sub-agent | `core:subagent` |
| Create or update a hook | `core:hooks` |
| Brainstorm an idea before planning | `core:brainstorm` |
| Plan a coordinated multi-artifact change (skill + sub-agent + hook) | `core:evolve` |
| Make a single git commit | `git:commit` |
| Open a pull request | `git:create` |
| Rebase the current branch onto main | `git:rebase` |
| Review PR comments / process review feedback | `git:review-comments` |
| Apply review fix decisions | `git:review-fix` |
| Review the current workspace diff | `git:review-diff` |
| List skills available in the current context (global / project / marketplace) | `core:skills-list` |

## When to invoke it

Trigger phrases:

- `/core:skills-suggest <intent>`
- `which skill should I use for X`
- `what's the right skill for…`
- `route this task`
- `recommend a skill`

The `<intent>` argument is required. If you omit it, the skill prompts you for it before proceeding.

## How it works

<Steps>
  <Step title="Read the intent">
    Takes `$1` as the intent argument. If empty, asks via `AskUserQuestion` for a one-sentence description of the task.
  </Step>
  <Step title="Match and rank candidates">
    Finds the rows in the Delegation map above whose "Intent / capability" cell best matches the intent (keyword overlap and semantic match). Caps at 3 candidates, ordered best-fit first.
  </Step>
  <Step title="Check local availability">
    Calls `core:skills-list` to get the full list of installed skills in the current context. Drops any candidate that isn't actually installed here.
  </Step>
  <Step title="Render the recommendation">
    For each remaining candidate, produces: the skill name, a one-sentence "why it fits", a copy-pasteable `Skill(...)` invocation with `args` filled from the intent, and a one-sentence boundary statement.
  </Step>
  <Step title="Handle no-match">
    If no canonical skill fits, says so explicitly and recommends `find-skills` (to install an external skill) or `core:evolve` (to create a new skill for a structural gap).
  </Step>
  <Step title="Handle overlap">
    If two candidates overlap significantly, surfaces the overlap explicitly with the boundary of each — never silently picks one.
  </Step>
</Steps>

## Output format

```text
Intent: `I need to find an existing component for a date picker`

**1. `explore:explore`**
- **Why it fits**: The codebase-exploration plugin includes a component-explorer specialist that surfaces existing UI components matching a description.
- **Invocation**: `Skill({ skill: "explore:explore", args: "find an existing date-picker component" })`
- **Boundary**: `explore:explore` reads the codebase to locate and describe components — it does not scaffold new components or edit existing ones.
```

## Boundaries vs similar skills

| Skill | What it does | What it does NOT do |
| :-- | :-- | :-- |
| `core:skills-suggest` | Routes an intent to an already-installed skill | Install or create skills |
| `find-skills` | Discovers and installs skills from the marketplace | Route tasks or create skills |
| `core:skill` / `core:evolve` | Creates new skills and artifacts | Route or install |
| `core:skills-list` | Lists installed skills | Recommend which to use |

## Anti-patterns

- Don't invent skill names. Only recommend skills confirmed available by `core:skills-list` AND present in the Delegation map.
- Don't install or create. This skill is strictly routing.
- Don't silently pick one candidate when two overlap — surface the overlap with boundary statements for both.
- Don't skip the availability check (step 3). Without it, you may recommend a skill the user doesn't have.

## Reference

- [`core:skills-list`](/plugins/core/skills/skills-list) — local availability check called in step 3.
- `find-skills` — for installing skills if no match exists.
- [`core:evolve`](/plugins/core/skills/evolve) — for creating skills if the gap is structural.
