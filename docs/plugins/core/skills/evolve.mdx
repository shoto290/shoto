`/core:evolve` is the coordination layer for multi-artifact changes. It inventories what already exists, proposes a dependency-ordered plan, gets your approval, then delegates each piece to the right executor. It never writes artifact files directly.

## When to use

Invoke `/core:evolve [capability description]` when you want to:

- Add a new capability that spans more than one artifact type (skill + sub-agent + hook).
- Get a structured plan before committing to any changes.
- Avoid duplicating functionality that might already exist.
- Orchestrate `skill-architect` and `subagent-architect` in the right order.

When invoked without arguments, `evolve` infers the capability from the current conversation â€” pain points, friction, repeated requests, or "I wishâ€¦" statements you've surfaced.

<Tip>
`evolve` uses `utils:explore` (the read-only codebase inventory from the `utils` plugin) to scan existing artifacts. This keeps the raw frontmatter dumps out of the main context window.
</Tip>

## How it works

<Steps>
  <Step title="Inventory (read-only)">
    Delegates to `utils:explore` to scan all skills, sub-agents, and hooks under `.claude/`. Returns a compact location-anchored report. No files are modified during this phase.
  </Step>
  <Step title="Classify">
    Maps your intent to one or more artifact types (skill, sub-agent, hook) and assigns an action: `create`, `update`, or `reuse`.
  </Step>
  <Step title="Surface conflicts and overlaps">
    Compares each proposed artifact against the inventory. If an existing artifact covers 70% or more of the need, `evolve` recommends `reuse` or `update` instead of `create`. All overlaps are called out explicitly.
  </Step>
  <Step title="Present the plan">
    Renders a dependency-ordered markdown plan: reused assets first, updates next, creates in dependency order (skill before the agent that wraps it, hook last). Includes a restart-required flag and test plan.
  </Step>
  <Step title="Get approval">
    Uses `AskUserQuestion` to confirm or adjust the plan. Never auto-executes. You can edit any entry before approval.
  </Step>
  <Step title="Delegate">
    Routes each approved entry: skills to `skill-architect`, sub-agents to `subagent-architect`, hooks to the `hooks` skill. Spawns architects in parallel when their work is independent.
  </Step>
  <Step title="Report">
    Summarizes what was created or updated. Explicitly flags which changes require a Claude Code restart.
  </Step>
</Steps>

## Delegation routing

| Artifact | Executor |
|---|---|
| Skill (create or update) | `skill-architect` sub-agent |
| Sub-agent (create or update) | `subagent-architect` sub-agent |
| Hook (create or update) | `hooks` skill via the `Skill` tool |

## Hard rules

- **Never writes artifact files directly** â€” always delegates to the appropriate executor.
- **Always inventories before proposing** â€” never guesses what already exists.
- **Prefers reuse or update** when overlap is 70% or more.
- **Always gets explicit approval** before delegating â€” no silent execution.
- Plans are ordered by dependency so the executor chain runs cleanly.

## Restart requirements

| Change | Restart needed? |
|---|---|
| New top-level skill directory | Yes |
| New sub-agent file | Yes |
| Hook change in `settings.json` | Yes |
| In-place edit to existing `SKILL.md` | No |
