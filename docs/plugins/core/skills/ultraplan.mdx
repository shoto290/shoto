---
title: "The `ultraplan` skill"
description: "Produce a codebase-grounded execution plan for a concrete code change, with every step anchored on a real local precedent."
---

`/core:ultraplan` writes a step-by-step execution plan for a single, concrete code change. It calls `explore:explore` to anchor every step on a real `path:line` precedent in your repo, then saves the finished brief to `.claude/plans/<slug>.md` ready to drive an implementation session.

## When to use

Invoke `/core:ultraplan [--mode quick|standard|deep] <goal>` when you want to:

- Turn a one-sentence goal into an ordered, verifiable action list before touching any code.
- Ensure every planned step mirrors a local convention rather than inventing a new pattern.
- Produce a durable plan file you can reference across multiple implementation sessions.
- Catch greenfield steps early — the `mirrors:` gate forces you to acknowledge when no precedent exists.

<Tip>
`ultraplan` is distinct from `core:evolve`. `evolve` is a meta-coordinator that decides which artifact types to create across the marketplace. `ultraplan` writes the direct execution plan once the artifact and scope are already known.
</Tip>

## How it works

<Steps>
  <Step title="Frame">
    Restates your goal as one crisp sentence and confirms the restatement. Resolves `--mode` (defaults to `standard`) and announces the active mode. This phase is never skipped.
  </Step>
  <Step title="Ground">
    Calls `explore:explore` in-skill via `Skill({ skill: "explore:explore", args: "..." })`. The number of parallel calls depends on the active mode. Returns `path:line`-anchored reports that feed the `## Reuse-first` section and every `mirrors:` slot in the plan.
  </Step>
  <Step title="Draft steps">
    Produces one step block per action needed to reach the goal. Each step carries `action`, `verify`, and `mirrors` fields at minimum. `deep` mode adds `risk` and `rollback` to every step.
  </Step>
  <Step title="Coherence check">
    Walks every step before writing. In `standard` and `deep`, any step with neither a real `mirrors:` anchor nor the escape hatch literal causes a re-draft. In `quick`, the same condition emits a warning but does not block.
  </Step>
  <Step title="Output">
    Computes a kebab-case slug from the goal, checks for filename collisions under `.claude/plans/`, writes the plan file, then prints the full plan inline so you have both the path and the content immediately.
  </Step>
</Steps>

## Intensity modes

The `--mode` flag controls how many explorer calls run, which step fields are required, and whether the mirroring gate is soft or hard.

| Mode | Explorer calls | Required step fields | Mirroring gate |
|---|---|---|---|
| `quick` | 1 | `action` + `verify` + `mirrors` | soft (warning only) |
| `standard` *(default)* | 2 parallel | `action` + `verify` + `mirrors` | hard (re-draft on failure) |
| `deep` | 4 parallel | `action` + `verify` + `mirrors` + `risk` + `rollback` | hard (re-draft on failure) |

<Note>
`--mode ultra` (all 8 explore specialists in parallel) is deferred to a future release. Passing `--mode ultra` routes you to `deep`.
</Note>

## The `mirrors:` slot

Every step must carry a `mirrors:` value. There are exactly two valid forms:

- A real `path:line-range` returned by the Phase 2 explorer calls — e.g. `plugins/core/skills/base/SKILL.md:12-18`.
- The escape hatch literal `mirrors: no precedent — creating new pattern` when the step is genuinely greenfield.

Any paraphrase of the escape hatch (`n/a`, `no example found`, `first of its kind`) fails the hard gate in `standard` and `deep`. This is intentional — the gate is the audit trail.

## Plan file schema

The output file at `.claude/plans/<slug>.md` follows a stable schema so other tools can parse it.

```text
---
name: <slug>
mode: <quick|standard|deep>
patterns:
  - anchor: <path:line-range>
    description: <one-line role>
created: <YYYY-MM-DD>
---

# Plan: <one-sentence goal>

## Context
## Reuse-first
## Steps
## Verification
```

Each step under `## Steps` is a `### Step N — <verb + object>` block with the fields required by the active mode.

## Arguments

`/core:ultraplan [--mode quick|standard|deep] <goal in one sentence>`

- `--mode` is optional. Omitting it selects `standard`.
- `<goal>` is required. If absent or too vague (one or two words with no scope), the skill asks one clarifying question before proceeding.

## Hard rules

- **Never skips Phase 2 grounding.** Even an obvious goal gets the explorer calls prescribed by the active mode. The value of `ultraplan` is anchoring steps on real precedents, not generating steps from memory.
- **Never invents `mirrors:` anchors.** Every `mirrors:` value is either a real `path:line-range` from a Phase 2 report or the exact escape hatch literal. Plausible-looking synthesized paths defeat the audit trail.
- **Never narrates internal reasoning in the plan file.** The brief is for execution. No "I chose X because…", no coherence-check recap, no commentary on explorer output.
- **No sub-agent wrapper.** v0 calls `explore:explore` directly in-skill. A dedicated sub-agent is deferred until a second consumer needs the same pattern-detection loop.
