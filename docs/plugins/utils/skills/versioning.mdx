title: "The versioning skill"
description: "Bump the semver version in a plugin manifest from Conventional Commits since the last release."
---

`/utils:versioning` reads the git log for a plugin directory, applies the Conventional Commits bump rules, and rewrites the `"version"` field in `plugins/<plugin>/.claude-plugin/plugin.json`.

## Overview

Every plugin in the shoto marketplace declares its version in its manifest. `versioning` keeps that version in sync with the commit history automatically — no manual edits, no guessing whether a change warrants a minor or a patch bump.

The skill uses the same `scripts/bump-plugin-version.sh` script that CI uses, so local and automated bumps are always consistent.

## When to use it

Load `/utils:versioning` when you or the user mentions:

- semver, semantic versioning, version bump
- releasing a plugin, cutting a release
- Conventional Commits, changelog generation
- bumping major / minor / patch for one or more plugins

## Usage

```
/utils:versioning [plugin-name]
```

If `plugin-name` is provided, only that plugin is bumped. If omitted, every plugin whose directory has commits since its last release tag is bumped.

```bash
# Bump the utils plugin only
/utils:versioning utils

# Bump all plugins with unreleased commits
/utils:versioning
```

## Conventional Commits mapping

| Commit prefix | Bump |
| :-- | :-- |
| `feat:` or `feat(scope):` | MINOR |
| `fix:` or `perf:` | PATCH |
| `feat!:`, `fix!:`, any `<type>!:`, or footer `BREAKING CHANGE:` | MAJOR |
| `chore`, `docs`, `refactor`, `test`, `style`, `build`, `ci` | none |

The **highest** bump kind across all qualifying commits since the last release wins. Major overrides minor overrides patch.

<Note>
  Plugins in this marketplace are pre-1.0 (`0.x.y`). The standard mapping still applies: `feat` = minor, breaking = major. This keeps behavior predictable now and correct after `1.0.0`.
</Note>

## Procedure

<Steps>
  <Step title="Fetch tags">
    Run `git fetch --tags` to ensure the local tag list is current.
  </Step>
  <Step title="Determine base ref">
    The base ref is:
    - The last tag matching `<plugin>-v*`, if one exists.
    - Otherwise `origin/main` when running on a PR branch.
    - Otherwise the repo root commit.
  </Step>
  <Step title="List qualifying commits">
    ```bash
    git log <base>..HEAD --format=%H -- plugins/<plugin>/
    ```
  </Step>
  <Step title="Compute and apply the bump">
    ```bash
    # Dry run — prints proposed bump without writing
    plugins/utils/skills/versioning/scripts/bump-plugin-version.sh <plugin> [--base <ref>]

    # Apply — rewrites "version" in the manifest
    plugins/utils/skills/versioning/scripts/bump-plugin-version.sh <plugin> [--base <ref>] --apply
    ```
    Without `--apply`, the script only prints `plugin=... current=... bump=... new=...`. With `--apply` and a non-`none` bump, it rewrites the manifest in place.
  </Step>
  <Step title="Validate the manifest">
    ```bash
    claude plugin validate
    ```
    Run from the repo root to confirm the manifest is still valid JSON with the correct schema.
  </Step>
  <Step title="Commit the bump">
    ```
    chore(<plugin>): release v<new-version>
    ```
    Use `chore` so the release commit does not itself trigger a future bump.
  </Step>
</Steps>

## Key behavior

- Reads `plugins/<plugin>/.claude-plugin/plugin.json` for the current version. Fails fast if the manifest is missing or the `"version"` field cannot be parsed.
- Determines the base ref automatically from git tags; falls back to `origin/main` or the root commit.
- Applies the highest bump kind across all qualifying commits — one `feat!:` overrides any number of `fix:` commits.
- Dry-run by default — `--apply` is required to write changes.
- The release commit uses `chore` to avoid triggering a recursive bump on the next run.
- A GitHub Action at `.github/workflows/version-bump.yml` runs the same script on every PR and commits the bump back to the branch automatically.

## Hard rules

<Warning>
  Never edit `plugins/<plugin>/.claude-plugin/plugin.json` version by hand if the versioning script will also run — you will create a conflict. Let the script own the version field.
</Warning>

<Warning>
  Confirm before applying a major bump. A major increment signals a breaking change; verify the triggering commit is intentional before running with `--apply`.
</Warning>
