title: "The `explore` skill"
description: "Explore a codebase area and return a compact, location-anchored report for downstream agents or your own context."
---

`/utils:explore` maps an unfamiliar area of a codebase without flooding your context with raw file dumps. It reads only what it needs, then returns a structured report every fact of which carries a `path:line` anchor.

## Overview

The skill runs in **fork mode** — it opens a fresh context window, does the file reading there, and delivers a single report back. Your main conversation stays uncluttered.

Three profiles let callers trade breadth against depth. The default is `deep-dive`.

## When to use it

Invoke `/utils:explore` when you or another agent needs to:

- Understand an unfamiliar module before editing it.
- Map entry points and dependencies before planning a refactor.
- Answer a narrow question about one symbol without opening every file manually.
- Feed a structured map to a downstream agent as its starting context.

Trigger phrases that load this skill: "explore", "map this codebase", "find entry points", "trace the flow", "what does X depend on".

## Usage

```text
/utils:explore [profile=survey|deep-dive|targeted] <topic or feature>
```

The `profile=` token is optional and must come first if present. Everything after it is the topic.

<Tabs>
  <Tab title="survey">
    ```
    /utils:explore profile=survey authentication
    ```
    Broad map. Opens at most **8 files**. Returns `## Entry points` and `## Dependencies` only. Use this when you need to know what exists before committing to a deeper look.
  </Tab>
  <Tab title="deep-dive (default)">
    ```
    /utils:explore authentication
    ```
    Full report. Opens at most **25 files**. Returns all four sections: `## Entry points`, `## Flow`, `## Dependencies`, `## Notes & gotchas`. Use this when you need to understand and reason about an area end-to-end.
  </Tab>
  <Tab title="targeted">
    ```
    /utils:explore profile=targeted why does requireAuth reject valid tokens after deploy?
    ```
    Narrow question. Opens at most **6 files**, all within two hops of the named target. Prepends a `## Scope` line, then emits only the sections the question actually requires. Use this for a single symbol or a specific bug.
  </Tab>
</Tabs>

## Profiles reference

| Profile | File budget | Sections emitted |
| :-- | :-- | :-- |
| `survey` | 8 | `## Entry points`, `## Dependencies` |
| `deep-dive` | 25 | All four sections |
| `targeted` | 6 (within two hops) | `## Scope` + only relevant sections |

Any unrecognised profile value falls back to `deep-dive`.

## Report format

The skill returns **only** the report — no preamble, no closing remarks. Every fact carries a `path:line` anchor so you can jump straight to the source.

```markdown
## Entry points
- src/auth/routes.ts:14 — `POST /login` — credential entry

## Flow
src/auth/routes.ts:14 → src/auth/service.ts:31 → src/auth/repo.ts:22

## Dependencies
- jsonwebtoken — src/auth/token.ts:1

## Notes & gotchas
- Tokens expire in 15 min; refresh is silent on requireAuth hit (src/auth/token.ts:58)
```

An empty section is written as `- (none)`. Sections the profile doesn't require are omitted entirely. Unconfirmed facts are prefixed with `?`.

## Key behavior

- Runs in fork mode — does not pollute the calling agent's context window.
- Uses `Glob` and `Grep` to locate relevant files before opening any.
- Stops exploring as soon as the required sections can be filled with confidence.
- Never invents facts — unknown values are marked `?` or omitted.
- Token-efficient by design: no code blocks quoting source, no narration.
- `profile=` parsing is strict: strip the token, use the remainder as the topic.

## Hard rules

<Warning>
  The report must not contain fabricated `path:line` anchors. Every anchor must correspond to a file the skill actually read during the current run.
</Warning>

<Note>
  The skill is allowed-tools restricted to `Read`, `Glob`, `Grep`, and `Bash`. It cannot write files or make network calls.
</Note>
