---
title: "The `explore-convention` skill"
description: "Map the naming patterns, folder structure, idioms, and error/logging/test conventions of a codebase area so new code blends in."
---

`/explore-convention` surfaces the implicit rules of a codebase area — what naming patterns, folder organization, idioms, and style conventions are in use — so you can write new code that fits. It runs in fork mode via the `convention-explorer` sub-agent and returns only the anchored report.

## When to use it

Trigger phrases that load this skill:

- `/explore-convention <area>`
- "how does this codebase handle errors"
- "what's the naming pattern here"
- "how are tests organized in this repo"
- "what idioms should I follow in module X"
- "how do we do logging here"

It is also dispatched automatically by `/explore` when your topic contains keywords such as `convention`, `naming`, `idiom`, `style`, `how do we`, or `error handling`.

## Usage

```text
/explore-convention [area=<path>] <topic>
```

The optional `area=<path>` prefix scopes the search to a specific directory. Everything after it is the topic.

## What counts as a convention

A pattern must appear in 3 or more distinct files to be reported. Patterns with fewer occurrences are treated as noise. When two competing patterns coexist, both are listed with a `?` prefix.

## What the skill captures

- **Naming patterns** — files, classes/types, functions/methods, variables/constants, test files.
- **Folder/file organization** — colocated tests, barrel files, module shape, one-component-per-folder rules.
- **Idioms** — language or framework preferences (e.g., `Result<T,E>` instead of throws, factory functions over classes, Zod at the boundary).
- **Style samples** — one anchor each for error handling, logging, and tests.

The sub-agent samples 8–12 files from different sub-folders to avoid bias toward a single module.

## Report format

```markdown
## Naming
- <kind: files | classes | functions | vars | tests> → <pattern> — sample at <path>:<line>

## Structure
- <folder or rule> → <role> — example at <path>:<line>

## Idioms
- <idiom> — sample at <path>:<line>

## Style
- error handling: <pattern> (<path>:<line>)
- logging: <pattern> (<path>:<line>)
- tests: <pattern> (<path>:<line>)
```

All four sections are always emitted. An empty section is written as `- (none)`. Conflicting conventions appear as two entries with a `?` prefix.

## Budget and rules

| Constraint | Limit |
| :-- | :-- |
| Files opened | 15 max |
| Files sampled | 8–12, from different sub-folders |
| Report length | 55 lines max |
| Anchors | Required on every fact |
| Unknown / conflicting | Prefixed with `?` |

## Sub-agent

This skill delegates to the [`convention-explorer`](/plugins/explore/agents/convention-explorer) sub-agent, which runs in a forked context. The report returned to you is the sub-agent's output verbatim.
