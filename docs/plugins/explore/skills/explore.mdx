description: "Orchestrator that routes a codebase exploration request to the right specialist skill, runs multiple specialists in parallel, or falls back to general mode."
---

`/explore` parses your request, picks the best specialist for the job ŌĆö architecture, component, flow, or convention ŌĆö and returns that specialist's report unchanged. For broad questions it dispatches multiple specialists in parallel and concatenates the results. When no specialist fits, it falls back to general mode and handles the request inline.

## Overview

The orchestrator never reads files itself in specialist mode. It delegates to a specialist skill (which runs its paired sub-agent in fork mode), then passes the report back to you verbatim.

In general mode ŌĆö when no specialist keyword matches ŌĆö it uses `Read`, `Glob`, `Grep`, and `Bash` inline and returns a canonical location-anchored report.

## When to use it

Trigger phrases that load this skill:

- `explore <any topic>`
- `map this feature`
- `find patterns in <area>`
- `is there a component for X`
- `trace the flow of Y`
- `what conventions does this repo follow`
- Anything that would benefit from a structured codebase map before acting

## Usage

```text
/explore [profile=<value>] <topic>
```

The optional `profile=` token, when given, overrides automatic routing:

| Value | Behaviour |
| :-- | :-- |
| `architecture` | Dispatch to `explore-architecture` |
| `component` | Dispatch to `explore-component` |
| `flow` | Dispatch to `explore-flow` |
| `convention` | Dispatch to `explore-convention` |
| `general` | General mode, `deep-dive` sub-profile |
| `survey` | General mode, broad 8-file budget |
| `deep-dive` | General mode, full 25-file budget (default fallback) |
| `targeted` | General mode, narrow 6-file budget around the named symbol |

## Intent detection

When no `profile=` is given, the orchestrator maps keywords to specialists:

| Keywords (English or French) | Specialist |
| :-- | :-- |
| pattern, architecture, layering, hexagonal, MVC, repository, factory, observer | `architecture` |
| reuse, existing component, duplicate, can I use, is there a, r├®utilis | `component` |
| flow, trace, by where, call chain, lifecycle, request path, ├®tapes | `flow` |
| convention, naming, idiom, style, how do we, error handling | `convention` |

- Exactly one match ŌåÆ dispatch to that specialist.
- Multiple matches ŌåÆ dispatch all matching specialists **in parallel** (single tool-use block).
- Zero matches ŌåÆ fall back to general mode (`deep-dive` by default, `targeted` for a specific symbol, `survey` for a bare module name).

Broad multi-word topics also trigger multi-specialist heuristics:

| Topic shape | Specialists dispatched |
| :-- | :-- |
| "understand `<area>`" | architecture + flow |
| "audit `<area>`" | architecture + convention |
| "before adding `<capability>`" | component + convention |

## Output contract

- **Single specialist** ŌĆö returns the specialist's report verbatim, nothing added.
- **Multi-specialist** ŌĆö concatenates reports with a `## architecture`, `## component`, etc. header per block, in dispatch order. No preamble, no closing.
- **General mode** ŌĆö returns the canonical location-anchored report for the chosen sub-profile (see below).

### General mode report format

```markdown
## Scope
<one line ŌĆö only emitted by the `targeted` sub-profile>

## Entry points
- <path>:<line> ŌĆö `<signature>` ŌĆö <role>

## Flow
<caller>:<line> ŌåÆ <callee>:<line> ŌåÆ ...

## Dependencies
- <external lib / service / DB table / API> ŌĆö used at <path>:<line>

## Notes & gotchas
- <constraint or non-obvious behavior> (<path>:<line>)
```

`survey` emits only `## Entry points` and `## Dependencies`. `targeted` prepends `## Scope` and emits only the sections the question requires.

<Note>
  General mode is handled inline by the orchestrator using `Read`, `Glob`, `Grep`, and `Bash` ŌĆö no sub-agent is spawned.
</Note>

<Warning>
  Every `path:line` anchor in the report must correspond to a file actually read during the run. No anchors may be fabricated.
</Warning>
