---
title: "Scope Hierarchy"
description: "Project vs user vs plugin scope for skills, sub-agents, and hooks — where artifacts live and which one wins when names collide."
---

Where you place an artifact determines who can use it and how it is discovered. Higher-priority scopes win when names collide.

## Skills

| Scope | Path | Applies to | Priority |
| :---- | :--- | :--------- | :------- |
| Enterprise | Managed settings | All users in the org | 1 (highest) |
| Personal | `~/.claude/skills/<name>/SKILL.md` | All your projects | 2 |
| Project | `.claude/skills/<name>/SKILL.md` | This project only | 3 |
| Plugin | `<plugin>/skills/<name>/SKILL.md` | Where the plugin is enabled | 4 (lowest) |

When names collide: Enterprise > Personal > Project. Plugin skills use a `plugin-name:skill-name` namespace and can never conflict with non-plugin skills.

Claude Code watches `~/.claude/skills/` and `.claude/skills/` for live changes. Adding or editing a skill takes effect within the current session. Creating a new top-level skills directory that did not exist at session start requires a restart.

Project skills also load from `.claude/skills/` in every parent directory up to the repo root, so starting Claude in a subdirectory still discovers root-level skills.

## Sub-Agents

| Scope | Path | Priority |
| :---- | :--- | :------- |
| Organization (managed) | Managed settings directory | 1 (highest) |
| Session-only | `--agents` CLI flag | 2 |
| Project | `.claude/agents/` | 3 |
| User | `~/.claude/agents/` | 4 |
| Plugin | `<plugin>/agents/` | 5 (lowest) |

**Project (`.claude/agents/`)** is the recommended default. Check it into git so the whole team gets the same sub-agents. Discovered by walking up from CWD to the repo root.

**User (`~/.claude/agents/`)** is for personal sub-agents available in every project.

<Warning>
Plugin sub-agents ignore `hooks`, `mcpServers`, and `permissionMode` — those fields are silently dropped. If you need them, copy the file into `.claude/agents/` or `~/.claude/agents/`.
</Warning>

`.claude/agents/` and `~/.claude/agents/` are scanned recursively, so you can organize sub-agents into subdirectories. The subfolder path does not affect the sub-agent's identity — only the `name:` frontmatter field matters. For plugins, a file at `agents/review/security.md` registers as `my-plugin:review:security`.

## Hooks

Hooks live at `.claude/hooks/` (project scope) or `~/.claude/hooks/` (user scope). Plugin hooks are declared in `<plugin>/hooks/hooks.json`.

The same priority order applies: project hooks and user hooks are processed before plugin hooks. Hooks from multiple sources are **merged** — all matching hooks fire, not just the highest-priority one.

<Note>
`--add-dir` paths grant file access but do not extend agent or skill discovery. To share sub-agents across projects, use `~/.claude/agents/` or a plugin.
</Note>
