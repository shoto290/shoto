Every decision in the shoto codebase — and in any Claude Code artifact built with it — must pass through these six principles. They define **what** to build; the [behavioral guidelines](/concepts/behavioral-guidelines) define how to approach the work.

<Note>
The SIMPLE framework was originally conceived by [pybuche](https://github.com/pybuche). It is adopted here as the foundation of the shoto stack.
</Note>

## The Six Principles

### S — Simple

Favor the simplest solution that solves the problem. Less code, fewer abstractions, no over-engineering. If a 200-line implementation could be 50 lines, make it 50.

### I — Intentional

Every line of code exists for a reason. No speculative features, no "just in case" logic. If you cannot articulate why a piece of code is there, remove it.

### M — Measurable

Changes must have observable impact. If you cannot verify the change works — through a test, a visible output, or an inspectable artifact — rethink the approach before writing code.

### P — Pragmatic

Ship what works today. Choose proven patterns over clever ones. Perfect is the enemy of done. A working solution delivered now beats an elegant solution delivered never.

### L — Layered

Build incrementally. Each change should be a stable, shippable layer on top of what already exists. Avoid big-bang rewrites; prefer additive, reversible steps.

### E — Envisioned

Keep the end goal in sight. Short-term decisions should align with the long-term product vision. A quick fix that forecloses a future capability is not pragmatic — it is shortsighted.

<Note>
SIMPLE defines **what** to build. The four [behavioral guidelines](/concepts/behavioral-guidelines) define **how** to approach the task.
</Note>
